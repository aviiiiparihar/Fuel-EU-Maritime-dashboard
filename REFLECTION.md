Building this FuelEU Maritime Compliance Dashboard with AI in the loop made the pace feel fundamentally different from a typical greenfield project. Instead of spending the first day on wiring and configuration, I could “declare” the stack and architecture and immediately get to a runnable scaffold. The biggest acceleration came from all the repetitive but necessary work: TypeScript strict configs, linting/formatting setup, Prisma models and repository skeletons, and the initial page/layout scaffolding on the frontend. Those are the sorts of tasks that are easy to get wrong in small ways and time-consuming to do by hand, but they’re also highly pattern-based—perfect for AI assistance.

Prompt-driven development also helped keep the project coherent. By writing requirements as prompts (routes, page behavior, repository ports, and constraints like “no external libs in core”), each step served as a small spec. That made it easier to build in layers: core domain first, then ports, then adapters, then UI. When something didn’t work, the prompt acted like a checklist of intended behavior rather than relying on memory.

The challenges were mostly in the “last mile” and integration edges. Supabase connectivity is a good example: the difference between direct connections and PgBouncer pooling matters for Prisma, and placeholder connection strings can silently derail everything until the first real query runs. Tooling and environment mismatches also showed up—Node/Vite version warnings, background processes locking folders on Windows, and type issues in chart tooltips. These aren’t hard problems, but they require careful debugging and often a few quick iterations.

If I were doing this again, I’d improve a few things up front. I’d document Supabase connection strings earlier (including `DATABASE_URL` vs `DIRECT_URL`) and add a simple health endpoint and CORS configuration from the start to make frontend-backend integration smoother. I’d also add a minimal set of end-to-end smoke tests (even one per feature area) to catch regressions quickly as the UI grows.

Overall, AI-assisted engineering feels most valuable when it’s treated as an amplifier: it generates strong defaults and boilerplate quickly, while a human still guides architecture decisions, validates assumptions, and resolves environment-specific issues. The best results come from iterative prompting—small, explicit steps with clear constraints—combined with continuous verification (build checks, runtime checks, and quick API calls). In that workflow, AI doesn’t replace engineering judgment; it removes the friction that often prevents good engineering from being applied consistently.
